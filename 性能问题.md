# 性能问题与优化记录（2026-05-01）

## 一、当前轮次卡死问题（已解决）

### 1) 问题现象
- 在“多个建筑物训练读条同时接近终点”时，游戏会在读条终点附近卡死。
- 卡死不依赖手动升星操作；即使只是正常训练流程，也会在训练完成瞬间触发。
- 复现特征是“同一时段多个建筑同时完成训练”，不是单个建筑稳定产兵时触发。

### 2) 最终根因
- **根因 A：训练完成同帧尖峰**
  - 多个建筑在同一帧触发 `produceXxx()`，会集中执行：
    - 对象池取对象/激活节点；
    - 星标同步；
    - 升星特效；
    - 目标点计算与移动初始化。
  - 这些操作叠加在同一帧造成主线程尖峰，导致卡死。
- **根因 B：0 间隔移动调度叠加**
  - 多个兵营在产兵后兜底路径使用 `schedule(moveUpdate, 0)`。
  - 同时产兵时会同时新增多个 0 间隔调度任务，进一步放大峰值。
- **根因 C：单位星标链路与产兵同帧叠加**
  - 单位常驻星标刷新与特效触发如果同步执行，会与产兵逻辑叠加，形成“终点帧拥塞”。

### 3) 已落地方案（当前稳定版本）

#### A. 产兵本身分帧（核心）
- 在 `TowerBuilder` 增加训练任务队列（`pendingUnitSpawnTasks`）。
- 各兵营读条完成后，不直接 `produceXxx()`，统一改为入队：
  - `WarAncientTree`（含补角鹰射手）
  - `HunterHall`
  - `SwordsmanHall`
  - `Church`
  - `MageTower`
  - `EagleNest`（含幼体成长后产成年体）
- `TowerBuilder.update()` 按预算消费训练队列，避免同帧激活多个单位。

#### B. 限速消费（进一步压峰）
- 训练队列消费从“每帧限量”升级为“**时间片限速**”：
  - `MAX_UNIT_SPAWN_TASKS_PER_FRAME = 1`
  - `UNIT_SPAWN_TASK_INTERVAL_MS = 80`
- 入队时按队列长度增加 `dueAtMs`，天然错峰执行。

#### C. 单位星标/特效分帧
- 在 `TowerBuilder` 增加单位星标任务队列（`pendingUnitStarOps`）。
- 训练后单位星标改为入队，不在产兵同帧强刷。
- 单位星标消费也改为时间片限速：
  - `MAX_UNIT_STAR_APPLY_PER_FRAME = 1`
  - `UNIT_STAR_TASK_INTERVAL_MS = 50`

#### D. 单位常驻星标轻量化
- `applyStarToUnitNode()` 改为单位独立轻量路径（就地复用容器），减少与建筑星池链路耦合。
- 单位星标尺寸改为 `0.8x`（`20 -> 16`），降低节点/渲染开销。

#### E. 兜底移动调度降频
- 各兵营产兵后兜底移动逻辑由 `schedule(moveUpdate, 0)` 改为 `schedule(moveUpdate, 0.05)`。
- 增加 6 秒超时退出，避免异常路径常驻调度器。

#### F. 对象池回收安全清理（防复用残留）
- `UnitPool.release()` 回收前递归停止 tween（含 `UIOpacity`）。
- 清理单位残留 `StarUpgradeFx`，避免复用后旧特效叠加。

### 4) 与“3星后放1星卡死”问题的关系
- 之前问题属于“对象池残留 + 松手多路径重入”叠加；
- 本轮问题属于“训练完成同帧峰值 + 0间隔调度叠加”；
- 两者可并存，但触发入口不同，已分别治理。

## 三、已恢复的历史“性能优化.md”内容（合并还原）

> 说明：原 `性能优化.md` 已被删除。以下内容已从历史备份/报告文档恢复并整理到本文件，作为统一归档。

### 1) 角色/敌人基类高频路径优化

#### 1.1 距离计算优化（平方距离）
- 大量 `Vec3.distance()` 替换为平方距离比较（避免频繁开方）。
- 适用于：
  - `Role` 系移动、避让、碰撞检测；
  - `Enemy` 系追击与碰撞判断；
  - 部分单位子类（如巨熊）状态判断。

#### 1.2 避让逻辑查表优化
- 对固定偏移角（如 `60/-60/90/-90...`）预计算 `sin/cos`。
- 运行时查表替代每帧三角函数计算，降低 CPU 热点。

#### 1.3 Vec3 临时对象复用
- 在 `Role` / `Enemy` / `Boss` 等热路径中复用 `tempVec3`，减少临时对象创建。
- 目标：降低 GC 压力，减少抖动帧。

### 2) 碰撞/检测频率优化

- 将部分高频检测从“每帧执行”改为“固定间隔执行”（例如 0.05s 或 0.5s）。
- 典型场景：
  - 碰撞检测节流；
  - 兽穴范围单位检测由逐帧改为间隔检测并缓存结果。

### 3) 产兵与行为初始化优化（新增并稳定）

- 训练完成逻辑分帧 + 限速（本轮核心）。
- 单位星标和特效分帧 + 限速。
- 产兵后移动兜底调度降频并超时退出。

### 4) 对象池与生命周期治理

- 建筑池/单位池复用前后统一重置关键状态。
- 清理残留节点（如星标容器、临时特效）和残留调度/tween。
- 避免“复用节点携带旧状态”导致异常行为或性能退化。

## 六、内存压测方法（人工）

以下流程用于在开发机上**主动加压**，观察长时间「全选回收 + 水晶升级 + 人口拉满」场景下的**内存占用**变化。压测前需**临时改代码**，测完请恢复，勿提交到正式版本。

### 1) 前置：降低水晶升级费用（仅压测）

- 在 `assets/scripts/role/Crystal.ts` 中，将 `getUpgradeCost(targetLevel: number)` 的返回值**临时改低**（例如固定为 `1` 或极小值），便于在第一关内频繁升级水晶、拉高人口上限，而不被金币卡住。
- **压测结束后务必改回原逻辑**，避免影响正式数值平衡。

### 2) 操作步骤

1. 进入游戏**第一关**。
2. **快速新建约 14 个建筑物**（按当前关卡可建造的兵营/建筑池操作即可，目的是尽快铺开产能）。
3. **持续升级水晶**，直到**人口上限至少为 24**（或场上可维持的满编人口达到压测所需规模）。
4. 在**人口已满**的情况下：
   - **不断全选场上单位**，并**点击回收**（或等价的全选回收操作）；
   - **同时继续升级水晶**（在费用已改低的前提下，可反复升人口/等级相关项）。
5. 以上组合操作的目的，是制造**大量单位销毁与再生成、UI 与选择状态频繁切换、复活/再部署**等压力。
6. **至少重复 5 次「全场复活」量级**的循环（即多轮「死光 → 再产满 / 再拉满人口 → 再全选回收」或项目内等价的**全场单位大规模更替**），每轮之间可稍作停顿以便在任务管理器或引擎 Profiler 中**采样内存**。
7. 压测过程中持续**观察进程内存占用**（Windows 任务管理器、或 Cocos/Chrome 远程调试等）。

### 3) 观察口径与经验预期

- 将游戏推进到约 **24 单位**规模（与人口/场上单位数一致）时，在**未出现明显泄漏、且对象池与回收链路工作正常**的前提下，相对进入关卡初期的基线，**正常内存增量约在 400 MB 左右**（随资源体量、编辑器/预览与真机差异会有浮动；该数值为**经验区间**，用于对比「异常持续爬升」）。
- 若在同规模下内存**持续单调上升、无明显平台期**，或**几轮全场复活后增量远超上述量级**，应结合 Profiler（堆快照、节点数、纹理/音频占用）排查泄漏或未及时释放的引用。
