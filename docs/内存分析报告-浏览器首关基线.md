# 浏览器首关内存分析报告（优化前基线）

**数据来源**：

- [tools/log/游戏优化前内存情况.txt](../tools/log/游戏优化前内存情况.txt) 中的 `[MEM_PROBE]` 日志  
- [tools/log/第一关-0-60秒-内存录屏.json](../tools/log/第一关-0-60秒-内存录屏.json)（Chrome Performance 导出）及同次录制的 **Performance 截图（0～约 84s）**

**场景**：第一关浏览器测试（Cocos 预览 `localhost:7456`）；任务管理器进程约上涨 400MB+（与下文 JS 堆指标需区分）  
**说明**：本文档仅作现状分析，**不包含代码改动**。

---

## 1. 结论概览

- 日志里 **`usedMB`（V8 已用堆）在约 95～143 MB 之间大幅抖动**，而 **`totalMB`（V8 已向 OS 申请的堆）从约 129 MB 缓慢涨到约 205 MB**，全程大约 **+76 MB 量级的堆提交增长**，且伴随 **GC 回收**（`usedMB` 会突然下降，`sinceStartMB` 甚至出现负值，属正常现象）。
- 任务管理器中的 **「进程约 +400MB」与 `performance.memory.usedMB` 不同步**：说明 **相当一部分增长不在 V8「used」统计里**，更可能落在 **GPU 显存（贴图、RenderTexture）、音频解码缓冲、Cocos/浏览器原生层、或其它非 `usedJSHeapSize` 内存**。当前埋点 **无法单独闭合解释这 400MB**。
- **强相关趋势**：**`sceneNodes` 从约 446 增到约 964（约 +518 个节点）**，与 **`totalMB` 爬升、飘字复用次数暴涨、多波次战斗** 时间线一致，表明 **场景内长期存活的对象图在变大**（含对象池挂在场景下的节点、场上单位与 UI 等）。

---
 
## 2. 指标怎么读（避免误读）

| 字段 | 含义与注意点 |
|------|----------------|
| `usedMB` | V8 当前使用的 JS 堆，**GC 会突然压低**，不宜单独作为「涨了多少」的唯一指标。 |
| `totalMB` | V8 **已提交**堆，相对平滑上升，更接近「堆这一层是否在持续扩容」。本基线约 **129 → 205 MB**。 |
| `sinceStartMB` | 相对开局 baseline 的 `usedMB` 差值，**GC 时会变负或乱跳**，不能当作单调曲线。 |
| `sceneNodes` | 当前场景 **递归节点总数**，本局 **约 446 → 964**，适合排查「谁在堆节点」。 |

若要对照任务管理器 **+400MB**，建议补充：**Chrome DevTools Memory / Performance、Heap Snapshot**，或引擎/原生侧 **GPU 纹理** 统计；仅靠当前 `[MEM_PROBE]` 不足以把「非 JS 堆」部分归因到具体模块。

---

## 3. 时间线与规模（Tick.summary 抽样）

从 `Tick.summary` 可见 **游戏时间约 25s → 200s（约 3 分钟多）**，波次 **约 2 → 8**，场上规模大致为：

- **`friendlyCount`**：约 **28 → 41**（建筑 + 训练单位增多）。
- **`enemyCount`**：多在十几到二十多（与波次、刷怪节奏相关）。
- **`sceneNodes`**：**约 648 → 900+**，与「内存体感上涨」方向一致。

说明：**单位与场景图复杂度随时间上升**；即使大量对象来自池子，**池内节点仍占用内存并计入节点数**。

---

## 4. 「大户」分级（基于本日志可支撑的部分）

### 4.1 第一梯队：敌人对象池 + 峰值时频繁 `instantiate`

日志中大量 **`EnemyPool.new`**，且多次出现 **`poolLenBefore: 0`**（当时池中无可用实例，只能 **新建**）。涉及 **`Orc` / `OrcWarrior` / `TrollSpearman`** 等，`active` 计数可到 **20+（Orc）** 等，说明：

- 战斗高峰时 **池被抽空 → 反复 `instantiate`**，对堆与节点数均构成压力；
- 波间敌人回池后，**池内会长期保留大量节点**（日志中常见 **`Dragon`、`OrcShaman` 等 pool≈10 且 active=0**），属于 **预热/池化的固定内存税**，抬高基线。

**判断**：敌人相关 **预制体体量 + 池规模 + 峰值同时存活**，是本日志里 **最能同时解释「堆提交上涨」与「sceneNodes 上涨」的一块**。

### 4.2 第二梯队：战斗飘字池（复用健康，峰值仍可观）

- `floatCreated` 从 **1 增至约 42**，`floatReused` 达 **1500+**，说明 **复用充分**，**不像「无限 new 飘字泄漏」**。
- 但 **`floatInUse` 一度到 26 左右**，`parentChildren` / `sceneNodes` 在飘字密集时冲高，说明 **瞬时并发飘字节点仍多**；对 **进程级 +400MB** 若为 GPU/贴图主导，则飘字可能为次要；若飘字节点绑重资源则需单独验证。

### 4.3 第三梯队：开局石墙连续 `BuildingPool.new`

日志开头 **连续多条 `StoneWall` + `新建对象`**，`active` 从 4 增至 14，对应 **开局批量铺石墙、池尚未预热满** 时的一次性 **`instantiate`**。主要影响 **开局前后一小段**，不太像整局 **+400MB** 的主因，但贡献 **早期堆与节点爬升**。

### 4.4 第四梯队：`UnitPool.new`（次数少、单点可能贵）

仅见少量 **`Priest` / `Hunter` / `ElfSwordsman`** 的 **`UnitPool.new`**，更符合 **人口上升后池扩容**。次数不多，**不像全局主因**；若单位预制体很重，单次 new 仍可能偏贵。

### 4.5 抽卡、小游戏、合并（本 log 未见「used 单调爆点」）

- **`BuffCard.show/close`**：`usedMB` 有波动，**未见**「每次关闭后 used 永久上台阶」的清晰模式；**不能**仅凭此断定抽卡 UI 泄漏，**不能排除** 贴图在 GPU 侧累积（当前未测）。
- **弓弦 / 磨剑**：各 **open + finalize** 一组，无单点爆炸，**偏正常 UI 生命周期**。
- **`Tower.mergeStar` / `Tower.place`**：与节奏一致，**未见**与 `totalMB` 台阶强绑定（**特效/音效** 若在其它通道占内存，当前未打日志）。

---

## 5. 与「进程 +400MB」的缺口（信息不足）

当前日志 **缺少**：

1. **GPU / 纹理 / RenderTexture** 占用（常为浏览器进程大头之一）。
2. **音频解码、分包 `assetManager` 缓存** 等用量。
3. **非 V8 的 ArrayBuffer / WASM / 原生插件**。
4. **两次 Heap Snapshot 的 diff**（可精确定位保留对象类型）。

因此：**可较有信心指出「场景节点与敌人/池化逻辑是 JS 侧主要关联项」**；**不能把 OS +400MB 全部归因到某一条 `[MEM_PROBE]`**。

---

## 6. 建议的下一轮取证（可选加日志 / 工具）

按优先级：

1. **Chrome**：同局录制 **Performance → Memory**，或在 **约 30s 与约 4min** 各拍一张 **Heap Snapshot**，对比 **Retained Size 前列构造函数/Detached DOM**。
2. **Cocos**：若 API 允许，低频打印 **`assetManager` 资源数量** 或在关键 **`resources.load` / bundle 加载完成** 处打 `[MEM_PROBE]`（含路径与引用计数思路）。
3. **`EnemyPool.release`**：在 **池满 `destroy`** 分支打计数/快照，判断是否 **「常满池 → destroy 再 new」** 导致堆与碎片问题（当前仅 new 侧较明显）。
4. **抽卡弹窗**：若 Heap 指向 UI/贴图，再在 **关闭后** 打一次 `snapshot`，确认是否释放。

## 8. 首关 0～60 秒 Performance / Trace 专项（补充）

本节对应 **同一次** 浏览器录制：Performance 内存曲线（截图）与 **`第一关-0-60秒-内存录屏.json`** 交叉核对。Trace 元数据时长约 **84.3s**（`range` ≈ 84 302 222 µs），与「约一分钟操作 + 导出窗口」量级一致。

### 8.1 截图现象（与 JSON 一致的部分）

- **JS Heap**：锯齿状 GC + **谷底整体抬高**（截图约 **28 → 64 MB**），符合「分配持续、回收不完全」形态。  
- **Nodes**：**阶梯式上升**（约 **1162 → 1367**），与此前 `[MEM_PROBE]` 里 **`sceneNodes` 随时间变大** 同一方向。  
- **Listeners**：在 **约 43s** 附近出现 **竖线式暴涨**（截图约 **55 → 1773**），与 Trace 中 **`UpdateCounters` 突变** 对齐（见下）。  
- **GPU Memory**：该段曲线 **相对平**，与「本段进程涨的主因不在显存曲线」不矛盾（进程总内存仍可能含其它项）。

### 8.2 Trace 中的量化锚点（`UpdateCounters`）

Chrome 在 Trace 里写入的 `UpdateCounters`（与 Performance 底部曲线同源）可读出：

| 阶段 | 约相对录制起点时间 | `documents` | `jsEventListeners` | `nodes` | `jsHeapSizeUsed`（约） |
|------|---------------------|---------------|---------------------|---------|-------------------------|
| 录制前半段（稳定期） | 0～约 41s | **3** | **102** | **1185** | ~34.3 MB（缓慢爬升） |
| **突变沿** | **约 42s** | **1** | **约 1697 → 1735** | **1288～1291** | **~54.2 MB**（阶跃） |
| 录制末段 | ~84s 前 | **1** | **55** | **1366** | **~44.3 MB** |

**解读要点**：

1. **`jsEventListeners` 在约 42s 与「分包资源加载」同一时刻阶跃**（见 8.3），**不是**均匀随战斗每帧缓慢漏出来的形态；更像 **一批 DOM/内部节点 + 调试/事件统计在短时间集中出现**，随后 Trace 末段 **listener 计数又回到 55**（与截图「之后一直顶在 1773」可能为 **DevTools 面板统计口径或不同次缩放/选中** 有关，以 **Trace 原始 `UpdateCounters` 为准** 更稳）。  
2. **`documents` 从 3 变为 1`**：常见于 **iframe / 辅助文档合并或卸载**（含预览、扩展、或引擎注入页结构变化），与「主文档上节点/监听重新计数」同时发生，**不宜直接当成游戏业务 bug 的单一证据**，但说明 **该时刻发生了较大的文档级变化**。  
3. **`nodes` 1185 → 1366+**：与 **懒加载预制体实例化** 一致，量级 **约 +180 个 DOM 节点**（Cocos Web 会把大量节点映射到 DOM），与 **分包加载后场上/池里对象变多** 相符。

### 8.3 突变时刻前后：Trace 里紧挨着的调用栈（根因方向）

在 **`jsEventListeners` 从 102 跳到 1697** 的 `UpdateCounters` 之前，同一主线程时间片上可见：

- **`XHRLoad` / `XHRReadyStateChange`**：`http://localhost:7456/query-extname/<uuid>`（预览服务器查询扩展名）。  
- 紧接着 **`ResourceSendRequest`** / **`XHRReadyStateChange`**：拉取 **`http://localhost:7456/assets/prefabs_sub/import/.../*.json`**（如 `2b596458-6189-49f5-b46f-217db08f56d5.json`、`b26fdc9c-8f8d-4342-a44e-7e6a782bfd37.json` 等）。  
- 栈顶落在 **Cocos 引擎 `bundled/index.js`** 的 **`downloadFile` → `downloadJson` → `process` → `load`/`fetch`/`async`** 链路上，即 **`assetManager` / 分包 `prefabs_sub` 的异步加载管线**。

**结论（本 Trace 能支持的）**：**约 42 秒处的内存与节点陡变，与 `prefabs_sub` 一批预制体/配置 JSON 的 XHR 加载及后续反序列化、实例化强相关**；你在这 60 秒内「几乎做完的操作」里，**若包含首次触发某波敌人/某类 UI 所需的分包资源**，时间线会与 **EnemySpawner 懒加载** 等逻辑吻合（具体需与游戏内操作时间点人工对齐）。

### 8.4 与第 4 节「大户」的关系（合并结论）

| 来源 | 与 8 节关系 |
|------|-------------|
| **敌人池 + 场上单位** | 分包加载完成后 **`EnemyPool.new`** 仍会反复出现（见原 `[MEM_PROBE]`）；Trace 的 **42s 阶跃** 更像是 **「资源到位 → 大量节点/监听一次性挂上」** 的 **门槛事件**，而非唯一长期泄漏源。 |
| **飘字池** | 本 Trace 未单独标记；长期战斗仍以 **复用 + 峰值 inUse** 为主。 |
| **抽卡 / UI** | 若发生在 42s 附近，可能与分包加载 **叠在同一窗口**；需用 **Heap 对比** 才能从 UI 对象里拆出占比。 |

### 8.5 JSON 文件体量与 Trace 摘要脚本

`第一关-0-60秒-内存录屏.json` 行数 **>100 万**，适合 **用 Chrome 重新打开分析**；全文不适合人工通读。

仓库内已提供 **流式解析**（不整文件 `JSON.parse`）的抽取脚本，生成摘要 CSV：

- **脚本**：`tools/scripts/extract-chrome-trace-key-events.js`  
- **用法**：`node tools/scripts/extract-chrome-trace-key-events.js tools/log/第一关-0-60秒-内存录屏.json -o tools/log/第一关-0-60秒-trace关键事件.csv`  
- **抽取**：`UpdateCounters`（仅在 `documents` / `jsEventListeners` / `nodes` 三者组合变化时打点，列中仍带当前堆占用）、`MajorGC` / `MinorGC`、`ResourceSendRequest`、`XHRLoad`（URL 含 `prefabs_sub`、`/assets/` 或 `query-extname`）。**不输出** `ResourceFinish`，避免上万行噪音。  
- **时间**：相对 Trace `metadata` 中 `initialBreadcrumb.window.min` 的 **秒**（微秒差除以 `1e6`）。
